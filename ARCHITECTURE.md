# EnglishPartner 架构说明

本文档说明 EnglishPartner 的系统结构、模块职责与设计原则。  
执行流程以 AGENTS.md 为准，工具调用以 TOOLS.md 为准，表达风格以 SOUL.md 为准。

---

## 1. 总体链路

EnglishPartner 的查询链路如下：

```text
用户输入
  ↓
OpenClaw Agent
  ↓
ep-query 统一入口
  ↓
项目 Python 虚拟环境
  ↓
query_engine.py / grammar_engine.py
  ↓
本地离线数据与语法规则
  ↓
Agent 组装学习结果
```

---

## 2. 设计原则

### 2.1 离线优先

单词释义、词根词缀、发音信息、语法规则优先来自本地数据和本地脚本。  
这样可以减少外部依赖，保持查询稳定。

### 2.2 统一入口

所有查询统一通过 `ep-query` 进入。  
这样可以避免模型直接调用脚本、直接读取数据文件，降低环境混乱和安全风险。

### 2.3 项目环境隔离

底层 Python 查询由项目自己的运行环境负责。  
模型不需要关心具体 Python 路径，也不应该绕过 `ep-query` 直接运行脚本。

### 2.4 数据与 Agent 解耦

Agent 负责理解用户意图、调用标准命令、组织学习结果。  
数据读取和规则匹配由底层查询模块完成。

---

## 3. 模块职责

### 3.1 OpenClaw Agent

负责：

- 判断用户意图；
- 区分完整查询和专项查询；
- 调用 `ep-query`；
- 生成 AI 例句；
- 组织最终回复。

Agent 不直接读取数据文件。

---

### 3.2 ep-query

负责：

- 提供统一命令入口；
- 分发 word / root / pronounce / grammar 查询；
- 调用底层查询脚本；
- 屏蔽 Python 环境细节。

---

### 3.3 query_engine.py

负责：

- 单词释义查询；
- 词根词缀查询；
- 发音查询；
- 查询结果格式化；
- 查询失败时返回空结果。

---

### 3.4 grammar_engine.py

负责：

- 本地语法规则匹配；
- 句型识别；
- 语法成分分析；
- 考点和难度输出。

---

### 3.5 本地数据

本地数据用于支持：

- 单词释义；
- 词性；
- 固定搭配；
- 词根词缀；
- 音标；
- 发音提示。

Agent 不直接访问本地数据，必须通过 `ep-query` 查询。
